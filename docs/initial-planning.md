# 智能支付宝场外基金策略系统（个人免费版）初步规划

> 目标：使用 **FastAPI + Render（后端） + GitHub Pages（前端） + Supabase（数据库）**，在免费前提下做一个仅供个人参考的基金策略系统。系统不做自动交易，不对接支付宝，仅输出建议。

## 1. 需求确认（本版）

你补充的关键输出已纳入：

1. **增减仓位建议**（例如：减仓 10%、保持不动、加仓 15%）
2. **信号强度**（例如：1-5 分，或 0-100 分）
3. **关键依据**（估值偏离、指数涨跌、波动率等）

技术约束：

- Python 依赖与环境统一用 **uv** 管理（不直接使用 pip）。
- Web 前端按 **GitHub Pages 推荐工作流**部署（GitHub Actions 构建 + 部署）。

---

## 2. 可行性结论（MVP 角度）

在“免费、单用户、非实时高频、仅策略参考”的范围内，方案可行。

- Render 免费层适合轻量 API 与定时任务，但有冷启动。
- Supabase 免费层足够存储策略参数、历史数据和每日信号。
- GitHub Pages 非常适合静态前端展示。

主要风险：

- 免费服务限额与休眠导致偶发延迟；
- 第三方数据源稳定性与字段漂移；
- 收盘前窗口（14:30~15:00）必须设计兜底机制。

---

## 3. 系统架构（MVP）

```text
[Data Source A/B] --> [FastAPI on Render]
                         |  \
                         |   -> 策略计算服务（信号、仓位建议、依据）
                         |
                         -> [Supabase Postgres]

[GitHub Pages Frontend] --> 调用 FastAPI API 展示今日建议与历史记录
```

### 模块拆分

1. **采集层**：抓取基金估值、指数当日变动（主源+备源）。
2. **策略层**：计算仓位调整、强度打分与关键依据。
3. **服务层**：FastAPI 提供查询、回溯与健康检查 API。
4. **展示层**：前端展示今日建议、历史曲线、解释信息。

---

## 4. 数据与策略输出设计

### 4.1 核心输出结构（建议）

`daily_signal`（可作为 API 响应与 DB 表结构基准）：

- `trade_date`: 交易日期
- `fund_code`: 基金代码
- `position_action`: `add` / `hold` / `reduce`
- `position_delta_pct`: 本次建议仓位变化百分比（-30 ~ +30）
- `signal_strength`: 信号强度（1-5 或 0-100）
- `confidence`: 置信度（0-1）
- `reasons`: 关键依据（JSON 数组，含指标与阈值）
- `data_timestamp`: 数据时间戳
- `source_summary`: 数据来源摘要

### 4.2 关键依据（首版建议）

- 基金估值相对近 N 日分位（高估/低估）
- 对应指数当日涨跌幅（市场环境）
- 近 N 日波动率（风险状态）
- 近 N 日最大回撤（风险约束）

示例（人类可读）：

- “当前估值处于近 60 日 18% 分位，低估；”
- “沪深300 当日 -0.9%，短期承压；”
- “波动率高于近 20 日均值，降低单次加仓幅度。”

## 5. 后端实现规划（FastAPI + uv + Render）

## 5.1 项目初始化（uv）

建议采用：

- `uv init` 初始化项目
- `uv add fastapi uvicorn pydantic sqlalchemy asyncpg supabase`
- `uv add --dev pytest ruff mypy`

说明：全流程使用 `uv run ...` 执行命令，不走 pip。

### 5.2 API 设计（MVP）

- `GET /healthz`：健康检查
- `GET /signal/today?fund_code=...`：获取当日建议
- `GET /signal/history?fund_code=...`：历史信号
- `POST /job/run-daily`：手动触发当日计算（可加 token）

### 5.3 调度策略

建议每日运行 3 次（交易日）：

- 14:30
- 14:45
- 15:00

用途：尽量捕获当天最新变化并修正最终信号。

---

## 6. 数据库设计（Supabase/Postgres）

首批表：

1. `fund_universe`：关注基金列表
2. `market_snapshots`：行情快照（原始数据）
3. `strategy_signals`：每日策略输出
4. `job_runs`：任务执行日志
5. `decision_logs`：最终展示给用户的建议记录

重点：

- 保留原始字段与来源，便于追溯；
- 策略结果和数据快照分离，方便复盘。

---

## 7. 前端部署规划（GitHub Pages 推荐工作流）

采用 GitHub 官方推荐路径：

1. 前端仓库（或 monorepo 子目录）构建静态文件。
2. 使用 **GitHub Actions** 构建并发布到 Pages。
3. 在仓库设置中将 Pages Source 设为 `GitHub Actions`。

前端建议：

- 可用 React/Vite 或纯静态页面。
- 使用环境变量注入后端 API URL。
- 页面重点：今日建议卡片、信号强度可视化、关键依据列表、更新时间。

---

## 8. 安全与合规边界

- 仅做“策略参考”，不做自动交易。
- 页面和接口显著标注“非投资建议”。
- 严禁在前端放 Supabase `service_role` 密钥。
- FastAPI 配置 CORS 白名单，仅允许 GitHub Pages 域名。

## 10. 需要你确认的意见收集（下一步完善依据）

请你按下面问题给偏好，我再输出 v2 细化方案（含表结构 SQL 和 API 契约）：

1. **基金范围**：你是只跟踪 1-3 只固定基金，还是维护一个候选池（例如 10-20 只）？
2. **仓位表达**：你希望输出“百分比变化”（+10%/-10%）还是“档位”（小幅/中幅/大幅）？
3. **信号强度**：更偏好 1-5 分（直观）还是 0-100 分（可量化）？
4. **策略风格**：更偏左侧（逆势低估加仓）还是右侧（趋势确认后加仓）？
5. **更新频率**：是否接受只在 14:45 与 15:00 两次更新，减少请求成本？
6. **前端技术**：你希望 React + Vite，还是先用纯静态 HTML（更轻量）？
7. **登录方式**：仅你一人使用，是否接受“简单访问口令”而不做完整账号体系？
8. **历史展示**：你是否需要“按月复盘图表”（收益曲线/回撤曲线）？
