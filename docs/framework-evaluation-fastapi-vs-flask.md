# 后端框架选型评估：FastAPI vs Flask（面向 best-select）

## 1. 背景与目标

当前项目已基于 `Python3.8 + uv + xalpha` 建立了 FastAPI 后端骨架。下一步需要判断：

- 是否继续使用 FastAPI；
- 是否切换到 Flask（或类似轻量框架）更合适。

本评估聚焦你的实际目标：

1. 单用户、免费优先；
2. 核心是策略接口与数据接入，而不是复杂 Web 后台；
3. 需要稳定可扩展（后续会接入任务编排、收益引擎、参数管理）。

---

## 2. 关键对比维度

| 维度 | FastAPI | Flask |
|---|---|---|
| 开发效率（API） | 高，内建类型校验和 OpenAPI | 中，需要手动补 schema / 校验 |
| 参数/响应约束 | 强（Pydantic 模型） | 弱（需自行约束） |
| 文档体验 | 自动生成 Swagger/ReDoc | 需插件或手工维护 |
| 异步能力 | 原生支持 async/await | 传统 WSGI 主导，异步支持路径更绕 |
| 生态成熟度 | 现代 API 场景强 | 历史悠久，插件广 |
| 学习曲线 | 中（类型与模型理念） | 低（上手快） |
| 中长期维护成本 | 低到中（契约清晰） | 中到高（容易散成“手写约定”） |

---

## 3. 在本项目中的真实影响

## 3.1 为什么 FastAPI 更贴合当前路线

1. **策略系统天然需要强契约**
   - `signal_score`、`action`、`delta_shares`、`reasons` 这类结构如果不做严格模型约束，后期极易漂移。
   - FastAPI + Pydantic 能把“文档即接口”落成可执行校验。

2. **后续会有多接口与运维接口**
   - 比如信号查询、收益查询、任务触发、参数管理等。
   - 自动 API 文档可显著降低联调成本。

3. **数据接入与任务编排会逐步异步化**
   - xalpha/外部源调用、批量基金计算、重试机制更适合保留 async 扩展空间。

4. **你现在已经有 FastAPI 骨架**
   - 切 Flask 的收益主要是“熟悉感”，但迁移成本客观存在，且对核心能力无明显增益。

## 3.2 Flask 仍然合适的场景

以下情况 Flask 可以更优：

- 团队全员对 Flask 极熟，且几乎不依赖自动 schema；
- 接口非常少、长期不扩展；
- 强调极简手写控制，不在意类型契约和自动文档。

当前项目并不完全符合这些前提。

---

## 4. 结论与建议

## 4.1 结论

**建议继续使用 FastAPI，不建议现阶段切 Flask。**

原因：

- 对策略系统而言，接口契约与可扩展性比“最小启动成本”更重要；
- 你已经有 FastAPI 起步代码，继续演进最稳；
- 后续的任务调度、参数治理、收益引擎会放大 FastAPI 的优势。

## 4.2 若你仍偏好 Flask，可采用折中方案

如果你非常想用 Flask，可采用“分层不变、框架可替换”的折中策略：

1. 保留 `data_sources/`、`services/`、`strategies/` 目录不变；
2. API 层单独抽象为 `interfaces/http/`；
3. 使用 contract tests（接口契约测试）约束响应结构。

这样即便未来切换框架，也不影响核心业务层。

---

## 5. 下一步落地建议（基于 FastAPI）

1. 增加统一响应 envelope（`code/message/trace_id/data`）。
2. 补齐 `GET /market/snapshot` 的异常映射（外部源失败 => 4xx/5xx + 错误码）。
3. 新增 `GET /signals/today` 的占位接口与模型。
4. 增加一组 API 契约测试（参数校验、响应字段、错误场景）。
5. 预留任务触发接口 `POST /jobs/run?session=pre_open|post_close`。

> 上述步骤不需要改框架，能直接推进你最关心的策略与数据接入工作。
